### New adapter — `src/adapters/html-scraper/asuncion-h3.ts` (mirror `onh3.ts`)

- **Filter:** keep only posts whose title matches `^\s*Run\s*#?\d+`i (all real runs do; `categories:[1]` is a secondary signal). Run # from the **title** via `extractHashRunNumber` or `/#(\d+)/`.

### 🔴 Embed-coord gotcha — `extractCoordsFromMapsUrl` does NOT match this form
Asunción start coords live in a Google Maps **embed iframe** `src`:
`https://www.google.com/maps/embed?pb=!1m18…!2d-57.60430085161278!3d-25.3020863951137!2m3…`
The embed `pb` order is **`!2d<lng>` then `!3d<lat>`** (lng first, lat second) and there is **no `!4d`**.
`src/lib/geo.ts extractCoordsFromMapsUrl` Pattern 1a requires `!3d…!4d…` and Pattern 1b requires `@lat,lng` — **neither matches**, so calling it on the embed URL returns `null` and the pin is silently dropped. Two options (pick one, add a unit test either way):
1. Extract in-adapter: `const m = embedSrc.match(/!2d(-?\d+\.\d+)!3d(-?\d+\.\d+)/); // [1]=lng [2]=lat` then build `{ lat: +m[2], lng: +m[1] }` and validate with `isValidCoords`. **Mind the order — 2d is longitude, 3d is latitude.**
2. Or extend `extractCoordsFromMapsUrl` with an embed pattern (preferred if reused) — but scope creep; in-adapter extraction is fine for a single source. Pass the parsed `{lat,lng}` straight onto the event; do NOT round-trip through `extractCoordsFromMapsUrl`.
