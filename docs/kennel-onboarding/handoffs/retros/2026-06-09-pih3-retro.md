## Handoff GAPS → research-prompt / process improvements (the actionable part)

### A. 🟢 There were essentially no handoff gaps — this is the template for a config-only HC repeat

Unlike Lisbon (which surfaced the two-part coord/location fix) or Madrid (the 11-year archive date mess),
Porto exposed **nothing new**. That is itself the signal: a 2nd-in-platform onboard that mirrors a
documented sibling should be a near-mechanical, zero-new-code change, and this one was. The handoff's
field-fill table, the recently-active rationale, the region-scope line ("Porto METRO only"), and the
inference-collision call were all pre-made and all held. Keep producing handoffs at this fidelity for
repeat-platform kennels.

### B. 🟡 The review-bot bare-`porto` suggestion was a FALSE positive — declined with evidence, not anchored

Gemini (medium) proposed broadening the inference rule to match bare `porto` via a negative lookahead
excluding Brazilian cities (`\bporto\b(?!\s+(?:alegre|velho|seguro|real|nacional))`). Empirically tested
before responding — it introduces **three new silent misattributions**: `"Porto Feliz, Brazil"` and
`"Porto Ferreira, Brazil"` (unlisted Brazilian Portos — the blocklist is necessarily incomplete) and
`"Porto-Novo, Benin"` (the hyphen isn't `\s+`, so the lookahead doesn't exclude it) → all wrongly inferred
Portugal. `COUNTRY_INFERENCE_RULES` is a heuristic *fallback*; the seed `country` field is authoritative
for real kennels, and PIH3's event text is "Porto, **Portugal**" which already matches the existing
`portugal` token. Trading one "bare Porto → USA" miss for three Brazil/Benin → Portugal corruptions is a
net loss. Declined on the thread with the empirical table and resolved.

> **Process note:** for country-inference review suggestions, test the proposed regex against the
> *collision* cases (other-country cities sharing the token), not just the happy path. A broadened
> matcher that silently misattributes is worse than a narrow one that falls through to the explicit seed
> `country`. Anchor on fail-loud-over-silent-corruption (the codex-oscillation memory), and bring the
> empirical table to the reply rather than asserting.

## Implementation / process learnings (loop context)

1. **🔴 The worktree-cwd trap DID bite this time — caught at diff review.** The initial seed/region `Edit`
   calls used **main-repo absolute paths** (`/Users/johnclem/Developer/hashtracks-web/prisma/...`) instead
   of the worktree path, so they landed in the *main checkout* (which was on a stale base with the user's
   uncommitted doc WIP). Caught it at the pre-commit `git diff` step (the worktree diff was empty;
   `git diff origin/main` in the main repo showed my edits tangled with origin/main being *ahead*). Recovery
   was surgical: confirmed the 4 files were clean at the main repo's HEAD before my edits (`git diff HEAD`
   showed only the pih3 additions), `git checkout HEAD -- <4 files>` to revert just those (the user's
   unrelated doc WIP untouched), then re-applied every edit to the worktree with worktree-prefixed paths.
   **Lesson (reinforces the worktree-cwd memory):** always `git status`/`git diff` in the worktree *before*
   staging, and prefix every `Edit`/`Write`/`rm` with the full worktree path — the Bash cwd resets to the
   main repo between calls, but the file tools take absolute paths verbatim.
2. **🟢 Post-merge ran from the worktree, NOT the main repo — because main had conflicting WIP.** The main
   repo (`/hashtracks-web`) was on an old commit (256dba7a, behind `origin/main`) with the user's
   uncommitted edits to `run-log.md` / `target-queue.md` / `source-platform-notes.md` — and those same
   files had *also* changed on `origin/main`, so a `git pull` would have clobbered the WIP. Instead, seed +
   `scrapeSource` ran from the **worktree** (its tree == `origin/main`) with the prod `.env` copied in
   temporarily (gitignored), then removed. The main repo's prod `.env` was never touched. `npx prisma
   generate` once per fresh checkout (gitignored `@/generated/prisma`). Node 25 again (no `fnm` on PATH; 25
   satisfies Prisma 7's "20+").
3. **🟢 Standalone tsx verify scripts need explicit `.env` loading.** The first prod-verify script connected
   to `localhost` (db `johnclem`) because importing `@/lib/db` does NOT auto-load `.env` outside Next.js —
   `set -a && source .env && set +a` before `npx tsx` fixed it (the script-env-loading memory). The `prisma
   db seed` itself was fine (loads `.env` via `prisma.config.ts`'s `dotenv/config`).
4. **🟢 Build the docs/retro PR in a clean worktree branch off `origin/main`** (the Lisbon retro's own
   process note). `origin/main` already had the merged onboard #2073, so the retro/run-log/queue edits sit
   cleanly on top with no conflict — and deliberately do NOT touch the user's stale main-repo WIP (which is
   superseded: `origin/main` already ships SHIPPED entries for Manila/Lisbon/Madrid that the WIP still shows
   as HANDED OFF).
5. **🟢 Live-verify proved END-TO-END before CI.** `adapter.fetch(source)` against the live HC API confirmed
   the 1-event count, the synthesized-vs-real title, the run-number passthrough, AND the coord drop — before
   tsc/lint/test (8800 tests green). Then a prod query after `scrapeSource` confirmed the canonical Event +
   re-geocoded Porto pin. The `after()`/`revalidateTag` "outside request scope" warnings from one-shot tsx
   are expected and harmless (DB writes persisted; the live page rendered #28 anyway).
6. **🟢 All gates green, no churn.** SonarCloud 0 new issues / 0 hotspots, Codacy 0, CodeRabbit pass — the
   data-only change (plus a 2-token inference regex) had no analyzer surface. No NOSONAR, no rewrites.

## TL;DR for the research prompt + platform notes

1. **A 2nd-in-platform config-only HC onboard should be near-mechanical** — mirror the nearest sibling
   (Lisbon here): GUID `publicKennelId` + `defaultTitle` + `staleTitleAliases`, `upcomingOnly` OMITTED,
   single `kennelCodes`, `trustLevel:8`, `scrapeDays:365`. Zero new adapter code; the Lisbon coord/sentinel
   work handles the city-level default pin automatically (`placeName===resolvable` → drop → re-geocode).
2. **New METRO under an existing COUNTRY = 2 region edits** (REGION_SEED_DATA record + `STATE_GROUP_MAP`)
   + optional inference token — NOT the full 5-edit new-country checklist. Do NOT re-add the COUNTRY /
   COUNTRY_GROUP_MAP / COUNTRY_CODE_TO_NAME (duplicate region records conflict at seed time).
3. **Country-inference: test review suggestions against COLLISION cases, not the happy path.** Bare `porto`
   misfires Porto Alegre/Feliz/Ferreira (Brazil) + Porto-Novo (Benin); `oporto|invicta` is the safe
   broadening. The seed `country` field is authoritative — a narrow fallback that misses is better than a
   broad one that silently misattributes.
4. **Recently-active framing is the antidote to "1 event = broken."** For a future-only feed (HC), a single
   upcoming event with a recent regular run (within ~2× the interval) is a healthy onboard, not a failure —
   state the cadence evidence in the handoff so the implementer doesn't second-guess a low count.
5. **Keep:** the verbatim-oracle capture (browser page-context when the sandbox can't resolve the Azure
   host) + the "reconfirm from local env" flag, the non-monotonic-run-number faithfulness, the
   NXDOMAIN-vs-live website DNS check (`pih3.run` dead → `ononinvicta.club`), the Porto-METRO-only region
   scope, and the bare-`porto` collision call — all landed first-try.
6. **Worktree discipline:** prefix every file-tool path with the full worktree path and `git diff` the
   worktree before staging — the Bash cwd resets to the main repo between calls, so main-repo-absolute
   `Edit`s silently land in the wrong checkout.
