## 🔴 The headline process failure: the backlog was invisible for six days

The daily runs kept producing handoffs, but **nothing ever committed them**. All six handoff files —
plus the run-log entries, four new platform notes, and queue rows 12–15 — sat **untracked/uncommitted in
the main repo working tree**. `origin/main` never moved. An onboarding session that branched from
`origin/main` therefore saw a handoffs folder that looked *complete*, and this session's first audit
confidently (and wrongly) concluded **"0 kennels left to onboard"** — while six live, verified kennels
sat unimplemented on disk. The user caught it ("there were some more onboarding docs added to the
folder"); a `diff` of the main-repo folder against the worktree surfaced all six instantly.

This is the **exact failure the 2026-07-05 backlog catch-up already diagnosed** ("`copy-newest-handoff.sh`
only ever surfaces the *newest* un-implemented handoff, so skipped days accumulate silently"), and whose
recommended fix — *"teach `copy-newest-handoff.sh` to warn when >1 un-implemented handoff exists (or add
an `--oldest` mode)"* — **was never built**. It has now recurred at 6× the scale (4 kennels then, 6 now).

> **Process change (do this one, finally):**
> 1. **Commit the daily run's output.** A handoff that only exists as an untracked file in one working
>    tree is invisible to every other checkout and to `git`. The daily run must commit + push its
>    handoff + run-log + queue + platform-note edits, or its work can silently rot. This is the root cause.
> 2. **Make the backlog loud.** `copy-newest-handoff.sh` should print a warning + count when >1
>    un-implemented handoff exists, and grow an `--oldest` mode.
> 3. **Audit against the source of truth, not a stale checkout.** The reliable audit is
>    per-handoff `kennelCode` → present in BOTH `kennels.ts` and `sources.ts`; run it against a synced
>    tree, and diff the main repo's folder for untracked handoffs first.

## Gaps the handoffs missed → prompt / platform improvements

### A. 🔴 The Divahhh Walkers/Runners pair would have silently lost a canonical

The handoff said of #39/#40: *"a Walkers/Runners split of the same slot → **keep both** (distinct
EventNumbers + titles, not a duplicate)."* Keeping both in the frozen JSON is **not sufficient**: both sit
on **2026-04-18 at 09:30**, and `merge.ts`'s `upsertCanonicalEvent` (~L1518) treats a same-date row as an
*update* when `looksLikeSameEvent` = `startTime` equal **OR** `runNumber` equal. Equal startTime → #40
would have silently overwritten #39, shipping 8 canonicals instead of 9 with no error.

Fixed per the established rule (memory: *same-day double-headers must differ on **both** startTime and
runNumber*): they already differ on runNumber, so the **Walkers row's `startTime` was cleared** (no
separately-published walkers' time) — both now survive as distinct canonicals. An automated
group-by-date collapse scanner was added to the extractor and reports **0 unresolved pairs** across all
four backfills.

> **Prompt change:** when a handoff says "keep both" for a same-date pair, it must also state **how** the
> two rows differ on startTime/runNumber — and if they don't, prescribe the defensible edit. "Keep both"
> alone is a silent-data-loss instruction.

### B. 🟡 "Drop the bleed hares" needs the actual strings, not just run numbers

`mapRunToRawEvent`'s shared `cleanHares` already drops TBA / "placeholder user" / `hares === location`,
which silently handled Rio #446 and Divahhh #36. But it can **not** catch bleed that merely *contains*
the venue (Poznan #12 `"Start point: Piwna Stopa, Brewery just south of Poznan Garbary…"`) or generic
placeholders (Divahhh #43 `"Multiple Hares"`, #44 `"Will be pulled from a hat!"`). Those needed an
explicit per-source rule.

> **Prompt change:** list the **verbatim hare string** to drop, not just "#12/#13 location-bleed" — and
> note which drops the shared cleaner already handles so the build doesn't write dead scrub code.

### C. 🟡 Douliu's title cleaning needed a fragment guard the handoff didn't anticipate

The prescribed rule (strip `^DH3\s*#?\d*`, bare-number → synth) is right but incomplete. Stripping
`"DH3 #160 in Douliu"` yields **`"in Douliu"`** — a location fragment that reads worse as a card title
than the synthesized `"Douliu H3 #160"`. Same for `"#178 with CBB & Hand Solo"` → `"with CBB & Hand Solo"`
(a hare fragment). Added a guard: a post-strip remainder opening with a preposition (`in|at|near|around|
with`) → `title: undefined` → merge synthesizes. Also had to strip a **bare** leading `#N` (the handoff's
regex required the `DH3` prefix, so `"#167 The "Peeping Hole" Run"` kept its number). Net: **14 genuine
themes kept, 17 synthesize**, embedded newline in #182 collapsed.

> **Platform note:** after a title-prefix strip, validate the *remainder* — if it isn't a theme (bare
> number, or opens with a preposition), clear it and let merge synthesize. Never ship a fragment.

### D. 🔴 Steel City's `BYDAY=4SA` fits neither ordinal — shipped flat instead

The handoff prescribed `FREQ=MONTHLY;BYDAY=4SA` ("4th Saturday"), citing 4 observed runs as "all
last/4th Saturday" — but those two descriptions **diverge in any 5-Saturday month**, and the observed
data splits across both:

| Run | Date | Saturdays that month | Ordinal |
|---|---|---|---|
| #4 | 2025-05-24 | 3, 10, 17, 24, **31** | 4th — **not** last |
| #5 | 2025-06-28 | 7, 14, 21, 28 | 4th = last |
| #6 | 2025-07-26 | 5, 12, 19, 26 | 4th = last |
| #7 | 2025-08-30 | 2, 9, 16, 23, **30** | **last (5th) — not 4th** |

So `BYDAY=4SA` would project 2025-08-23 and miss the real 08-30 run; `BYDAY=-1SA` would miss 05-24.
**Neither ordinal is correct.** Because `scheduleRules` are HIGH-confidence and override the flat fields,
this would feed Travel Mode a wrong date roughly whenever a month has five Saturdays — and the weekly
rule-drift detector is **weekday**-based, so an ordinal error is invisible to it. Shipped **flat fields
only** (Monthly / Saturday / 12:00), letting Pass 2 derive a LOW-confidence rule — the Algarve
("seasonal shift → a single RRULE would misrepresent it") and Lune Valley ("shipped flat, skipped the
optional BYSETPOS rules") precedent. Revisit once HC accumulates real history for the kennel.

> **Prompt change:** "all last/4th Saturday" is not a cadence — those are two different RRULEs that only
> coincide in 4-Saturday months. Before proposing an ordinal `BYDAY=NSA`, test the claim against every
> observed date **in 5-Saturday months specifically**; if they split, ship flat.

### E. 🟡 `HarrierCentralConfig` has no title-strip knob → past/future title divergence

Douliu's title cleaning lives only in the backfill, so its 31 past runs get cleaned/synthesized titles
while a future live run would keep HC's raw `eventName` (e.g. a bare `"206"`). There's **no fingerprint
risk** (backfill is past-only, the adapter is future-only — they never touch the same event), but it is a
cosmetic inconsistency. The handoff floated `titleStripPatterns` as a "consider"; the config type doesn't
support it. Deferred, not silently dropped.

> **Follow-up:** if bare-number HC titles recur, add `titleStripPatterns` to `HarrierCentralConfig` so
> the live adapter and the backfill share one cleaning rule.

### F. 🔴 A targeted `seedKennels` post-merge does NOT create ScheduleRule rows

`runScheduleRuleBackfill` (Pass 3, which is what reads `KennelSeed.scheduleRules`) is invoked **only from
a full `prisma db seed`** — `prisma/seed.ts:755`. But the standing post-merge runbook deliberately uses a
**targeted `seedKennels` subset** to avoid a full seed reverting other sources' prod config (barbados
#2471). So a new kennel's `scheduleRules` would be **silently dropped**: kennel + source rows land, zero
ScheduleRule rows, Travel Mode falls back to LOW-confidence sentinels, and nothing errors.

This batch ships 4 rules across 2 kennels (mh3-gb ×3, douliu-h3 ×1), so the post-merge step must run a
**scoped** Pass 3 — the Taoyuan Metro precedent (never the global, which touches all 462 rules and runs
`deactivateStaleRules`):

```ts
const mine = KENNELS.filter((k) => ["mh3-gb", "douliu-h3"].includes(k.kennelCode));
const planned: PlannedRule[] = [];
await runKennelSeedPass(prisma, planned, {}, mine);  // 4th arg scopes the pass
await applyUpserts(prisma, planned);                 // NOT runScheduleRuleBackfill
```

> **Runbook change:** any handoff whose kennel carries `scheduleRules` must spell out the scoped Pass-3
> command in its post-merge section. "Targeted `seedKennels`" alone silently loses them.

2. **🟡 `titleStripPatterns` on `HarrierCentralConfig`.** Douliu's bare-number titles can't be

   one to the HC config would let live + backfill share a single rule. Blast radius today is one

## Post-merge incident — importing `prisma/seed.ts` auto-runs the FULL seed

The post-merge runbook says "targeted `seedKennels`". Reusing the *real* tested `seedKennels` means
importing it — but **`prisma/seed.ts` has an unguarded top-level `main().catch()`**, so
`import { seedKennels } from "./prisma/seed"` fired the entire full seed as an import side effect,
running concurrently with the scoped call against prod. The tell was a string of misleading `P2002`
"unique on (name)" errors on region names that didn't exist (two racing `create`s) — which looked like
a driver / prepared-statement bug and cost real time to chase before the double `main()` was spotted.

The scoped one-shot's `process.exit(0)` killed the runaway `main()` right after region creation, so the
damage was near-zero (only the 6 batch kennels touched; no other configs; `deactivateStaleRules` never
ran). But the trap is real and will recur.

> **Fix ([PR #2655](https://github.com/johnrclem/hashtracks-web/pull/2655)):** add the entry-point guard `scripts/backfill-schedule-rules.ts`
> already uses — `const entryPoint = (process.argv[1] ?? "").replace(/\\/g,"/"); if
> (entryPoint.endsWith("/seed.ts")) main()...` — so importing `prisma/seed.ts` is inert and only
> `npx prisma db seed` / direct exec runs it.
>
> **Rule for the runbook:** a scoped post-merge seed must **never** `import` from `prisma/seed.ts`.
> Write self-contained upserts against the `@/lib/db` singleton, and for scoped Pass 3 import from the
> already-guarded `scripts/backfill-schedule-rules.ts` (`runKennelSeedPass` + `applyUpserts`). Read the
> prod `DATABASE_URL` from the **main repo** `.env` (the worktree's may point at the local dev DB) and
> pass it explicitly with a host guard.
