## Handoff GAPS → research-prompt / platform-note improvements (the actionable part)

### A. 🟡 "Drop the default pin" must ALSO suppress the placeholder string from `event.location`

The handoff framed the open coord decision as binary — **(a) accept the Lisbon-centroid pin** or **(b)
extend the geocode-fail sentinels to drop coords**. Choosing (b) and stopping at coords is a **half-fix**:
`hcGeocodeFailed` only gates the lat/lng, but the event's *location text* is composed independently by
`composeHcLocation`, whose `stripTba` only drops "TBA" (not "TBD" / "No location provided" / "ANNOUNCED
LATER via Hares"). So coords get dropped while `event.location` still stores and the merge pipeline
geocodes the placeholder as meaningless text (Codex P2 caught this in review — `sanitizeLocation` in the
merge path only filters generic TBA/TBD). The complete fix treats the two as one decision: a shared
`GEOCODE_FAIL_SENTINELS` set drives BOTH paths — `hcGeocodeFailed` (drop coords + `dropCachedCoords`) AND
`composeHcLocation` via `stripPlaceholderLocation` (drop the text so the row renders unlocated). Result:
26/27 unlocated, 0 fake pins.

> **Prompt / platform change:** for an HC kennel whose venues are all placeholders, "drop the default
> pin" is a TWO-part change — drop the coords AND suppress the placeholder text from `event.location`
> (otherwise the merge stores/geocodes the placeholder string). Present them as one decision, not just
> the coord half. Generalizes the `cleanLocationName` / `dropCachedCoords` memories to HC's two-field
> location model.

### B. 🟡 Geocode-fail sentinels are a SHARED HC concern — seed the set wider than one kennel's strings

LH3 only uses three placeholder strings, but the same pattern recurs across HC kennels (UK/Ireland hashes
use "TBC" / "to be confirmed" / "to be announced" ubiquitously). Gemini + claude-review both flagged
pre-seeding these. Adding them to the shared `GEOCODE_FAIL_SENTINELS` set is zero-risk (exact-match, set
lookup) and means the next HC kennel with placeholder venues needs no adapter change at all.

> **Prompt change:** when adding an HC placeholder sentinel, add the common family
> (`tbd`/`tbc`/`to be determined`/`to be confirmed`/`to be announced`/`no location provided`), not just
> the one kennel's literal — it's a shared adapter concern, and the set is the config-free path for future
> HC kennels.

### C. 🟢 The `composeHcLocation` "Portugal"-as-location edge is CORRECT behavior, not a miss

One event (#1025 "Pan Iberian Hash Weekend") legitimately carries `location:"Portugal"` — a real, if
coarse, location, NOT a placeholder sentinel. The exact-match sentinel set leaves it intact, which is
right (a country name geocodes to a country centroid; better than nothing). Worth stating so a future run
doesn't "fix" it into a sentinel.

> **Platform note:** sentinel suppression is EXACT-MATCH only — coarse-but-real locations ("Portugal", a
> bare city name) must survive. Don't broaden the set to substring/prefix matching.

## Implementation / process learnings (loop context)

1. **🟢 No worktree-cwd trap this time — clean worktree off `origin/main` for the docs PR.** The onboarding
   PR ran from the standard merged feature branch; the docs/retro PR is built in a **fresh worktree off
   `origin/main`** (which already had the concurrently-merged Madrid #2034), so #2034's changes are
   "pulled in" automatically with zero conflict. The original onboarding worktree had no `.env`, so all
   post-merge steps (seed, `scrapeSource`) ran from `/hashtracks-web` on `main` (prod `.env` → Railway).
   `npx prisma generate` once per fresh checkout (gitignored `@/generated/prisma`). Throwaway tsx scripts
   live **inside** the repo (the `@/` alias doesn't resolve from `/tmp`). Node 25 again (no `fnm` on PATH;
   25 satisfies Prisma 7's "20+").
2. **🟢 `composeHcLocation` is decoupled from `hcGeocodeFailed` — verified before extending.** Reading the
   adapter confirmed `composeHcLocation` derives its own place/address logic and never calls
   `hcGeocodeFailed`, so extending the geocode-sentinel set could NOT silently change venue text — the
   text suppression had to be added explicitly via `stripPlaceholderLocation`. Confirming the coupling
   before editing avoided both an over-broad change and a missed half.
3. **🟢 Review bots earned their keep — and were RIGHT, not oscillating.** Codex's P2 (placeholder text
   still leaking) was a real, verified gap I'd flagged myself in the ship summary; Gemini/Claude's sentinel
   expansion was a low-risk generalization. Both were implemented with tests rather than anchored against.
   The remaining bot comments (CodeRabbit/Qodo rate-limit notices, the `"Portuguesa"` demonym-alias nit)
   needed no action. SonarCloud passed clean (0 new issues / 0 hotspots) — the set-lookup-not-regex choice
   pre-empted the usual S5852/S5843 fights.
4. **🟢 Live-verify proved the END-TO-END behavior, not just unit tests.** A throwaway
   `adapter.fetch(source)` against the live HC API confirmed the 27-event count, the synthesized titles,
   AND the new coord/location suppression (26/27 unlocated, 0 pins) — before tsc/lint/test. Unit tests
   alone wouldn't have shown the "Portugal"-survives edge.
5. **🟢 Post-merge spot-check against `hashtracks.xyz`** (NOT `.com` — parked-for-sale, per the Madrid
   retro) confirmed the page renders the full #1016→#1042 schedule. `scrapeSource`'s post-write `after()`
   IndexNow/`revalidateTag` "outside request scope" warnings via one-shot tsx are expected and harmless
   (DB writes already persisted).
