3. The kennel `country` was set to the full name **"United Arab Emirates"**, not the handoff's `"AE"` —
   every non-US kennel uses the full country name; `AE` belongs only in `region.ts`.

## Deviations from the handoff (each driven by live data)

### A. 🟢 Kennel `country` = "United Arab Emirates", not "AE"
Every non-US kennel in `kennels.ts` uses the full country name (`"Sri Lanka"`, `"Thailand"`, `"UK"`); none
uses a 2-letter code. `AE` belongs only in `region.ts` (`abbrev` + `COUNTRY_CODE_TO_NAME`). One-line fix,
caught by reading the existing convention before trusting the handoff snippet.

### B. 🔴 The real schedule is a Mon/Sun day-alternation → drove the mixed-cadence FEATURE (#2310)
The handoff (and the upstream site copy) framed it as "every Monday, except the odd Sunday afternoon." The
live Hare Line (49 dated runs) showed **Monday 29 / Sunday 20 over 12 months, interleaved with no clean
season boundary** — a weekly kennel whose *run-day itself* alternates, with Sunday start-times sliding
14:00–18:00 by daylight. The handoff's `FREQ=MONTHLY;BYDAY=SU` (no ordinal) would expand to every Sunday →
~2 projections/week vs ~1 actual (the #2250 ledger-pollution anti-pattern).

The schedule model couldn't express this: flat fields apply one frequency to all days; seed `scheduleRules`
were hard-pinned HIGH and opt the kennel out of Pass-2 derivation. **#2294 interim** shipped flat fields
(→ Pass-2 MEDIUM Monday) + the Sunday in `scheduleNotes`, and a written **proposal doc**. **#2310** then
built the capability: optional per-rule `confidence`, Pass-3 acceptance of `CADENCE=…` sentinels (stored
verbatim — never `normalizeRRule`, which reorders `CADENCE` to the tail and breaks the engine's `startsWith`
checks), `explainSentinel` `CADENCE=WEEKLY` → "Sometimes runs on Sundays". Re-seeded `[MEDIUM Monday, LOW
`CADENCE=WEEKLY;BYDAY=SU`]` — Monday is **MEDIUM** (plurality, not every week — HIGH would overstate it);
the Sunday is a LOW sentinel → `date=null` possible-activity, **never snapshotted** (`prediction-ledger.ts`
skips `!proj.date`), so it adds **zero** false MISSes to precision.

> **🟢 Prompt/platform-note add:** for a kennel whose **run-day alternates** between two weekdays, pull the
> real weekday distribution from history first — don't trust "occasional X" framing. Model the dominant day
> as a MEDIUM/HIGH dated rule and the secondary day as a **LOW `CADENCE=WEEKLY;BYDAY=XX` sentinel**
> (possible-activity, never dated/snapshotted). NEVER a second un-ordinal `FREQ=MONTHLY;BYDAY=XX` HIGH rule
> (over-projects every week — #2250). Memory: `project_mixed_weekday_cadence_prediction_followup`.

### C. 🔴 Backfill upgraded to the full pre-2021 venue archive (151 runs) — then the orphan trap (#2304)
The handoff rated history "shallow/low-value, optional." But the recent ~50 Hare-Line runs ingest on the
adapter's own first scrape anyway, so the *real* value-add is the pre-2021 `?cat=3` WP-post era, which
**carries venues in the post title** ("Run 2202 – 8th August 2021 – Trade Centre Apartments"). Backfilled
**151 venue-bearing runs (2018–2021)**; the stray sub-series the guard excluded turned out to be the
COVID-era **"Virtual DH3" #1–22** — exactly the "Virtual DH3" the handoff said to keep out.

The trap: the first backfill (mirroring `bfmh3`) inserted RawEvents via `prisma.rawEvent.createMany(
{processed:false})` — which does NOT create canonical Events; "trigger a scrape to merge them" is wrong
(`scrapeSource` only merges the live adapter's fetch). 151 orphans. **#2304** rewrote it to
`runBackfillScript`→`reportAndApplyBackfill`→`processRawEvents`, which creates Events too and **adopts**
existing orphans (`handleDuplicateFingerprint`→`false` on `processed:false`+`eventId:null`, logs
`branch=adopt-orphan`) — re-running healed the 151 without deleting. Memory:
`reference_backfill_must_route_through_merge_pipeline`.

## GAPS / corrections → research-prompt / platform-note improvements

### D. 🔴 MEC's JSON-LD `@type:Event` time is BUGGY — parse the displayed time, not JSON-LD `startDate`
The handoff flagged the JSON-LD island as "unverified (web_fetch strips `<script>`)." It IS present on the
home card — but its `startDate` (`2026-06-29T23:00:00+04:00`) is the run at **19:00 local** mis-encoded
(local clock emitted as a UTC instant then tagged with the local offset → off by the offset). Trust the
displayed `.mec-event-time` / `.mec-start-time` `HH:MM` instead; the JSON-LD date is fine, the time is not.
MEC also renders agenda rows in **duplicate skins** (≈3× per run) → dedup by run number (parsed from the
title text, not the `?mec-events=…` slug, which gets `-265`-style collision suffixes). Memory:
`reference_mec_modern_events_calendar_adapter`.

### E. 🔴 Schedule-rule DATA changes need a companion migration — Vercel runs `migrate deploy`, never `db seed` (#2310, Codex P2)
The mixed-cadence *code* ships on deploy, but the *data* (dh3-ae's two ScheduleRule rows) only reaches prod
via `db seed`, which Vercel never runs — so a plain deploy would keep the old single Pass-2 Monday rule and
the feature would silently not ship. Added an idempotent DATA migration (mirroring the #2270/#1724/#1723
schedule-rule corrections) that upserts both rows; **Vercel's `migrate deploy` applies it on the preview
build**, so the rules hit prod automatically. Verified the SQL against prod in a **ROLLBACK transaction**
(`INSERT 0 2`, correct rows, rolled back) before committing, and confirmed `_prisma_migrations` recorded it
post-merge. Memory: `reference_kennel_seed_null_fill_skips_changes`,
`reference_vercel_preview_applies_migrations_to_prod`.

### F. 🟡 Change 2 (recall) deferred — recall isn't implemented yet
The proposal's "exclude sentinel-covered weekdays from the recall false-negative denominator" targets a
recall metric that **doesn't exist**: `score-prediction-ledger.ts` computes precision only and flags recall
as a deferred follow-up. Precision is already protected by the LOW-sentinel/`date=null` path, so building a
speculative recall computation would be out of scope. Left a forward note in `score-prediction-ledger.ts`
for whoever builds recall (treat a real event on a sentinel-covered weekday as "cadence-covered", not a
false negative).

### G. 🟢 `FREQ=LUNAR` excluded from Pass-3 sentinels (claude review on #2310)
`isCadenceSentinel` initially blessed `FREQ=LUNAR` too; removed it — lunar cadences need phase+timezone
metadata not representable on a seed `scheduleRules` entry, so they belong on a STATIC_SCHEDULE source. Pass
3 keeps rejecting `FREQ=LUNAR` from seed rules (matching the `KennelScheduleRuleSeed` doc); Pass 1's real
lunar rules from STATIC_SCHEDULE sources are unaffected. Narrowed the classifier to
`/^CADENCE=(BIWEEKLY|MONTHLY|WEEKLY)\b/`, aligned with `CADENCE_EXPLANATIONS` in `projections.ts`.

### H. 🟡 Deferred follow-up: directory weekday filters ignore sentinels (Codex P2 on #2310)
`collectKennelWeekdays` derives directory/region weekday facets from `parseRRule`, which rejects `CADENCE=…`,
so dh3-ae won't appear in a "Sunday kennels" filter (only its parseable Monday rule surfaces it). No
regression (the Sunday was never in the filter); it's a separate surface + a UX call (should LOW *hints*
drive hard filters?). Spawned as a tracked follow-up rather than scope-creeping the prediction PR.

   — both fixed by extracting/data-driving), S8786 ReDoS on a `\s*`-adjacent-to-alternation split (→ literal

   the dh3-ae rules to prod as soon as the PR's preview built, before merge. Dry-run the SQL in a ROLLBACK
   txn first (it's immutable + hits prod early). `reference_vercel_preview_applies_migrations_to_prod`.
3. **🔴 Post-merge prod steps need explicit user authorization** (recurring) — the additive `db seed` +
   backfill ran on "PR merged", but each was confirmed; prod host is `www.hashtracks.xyz`.

   proxy without `BACKFILL_ALLOW_SELF_SIGNED_CERT` (`ssl:undefined` non-prod), unlike `createScriptPool`.
5. **🔴 `vitest.config.ts` excludes `**/.claude/worktrees/**`** → ran via a disposable in-worktree

## TL;DR for the research prompt + platform notes

1. **🔴 Day-ALTERNATING kennel (run-day flips between two weekdays) → pull the real weekday split from
   history; model dominant=MEDIUM dated + secondary=LOW `CADENCE=WEEKLY;BYDAY=XX` sentinel.** Never two HIGH
   weekly rules and never an un-ordinal `FREQ=MONTHLY;BYDAY=XX` HIGH (over-projects every week). The
   mixed-cadence capability now exists (#2310: per-rule `confidence` + Pass-3 `CADENCE=…` acceptance).
2. **🔴 Backfill MUST route through `runBackfillScript`→`processRawEvents`** — direct
   `rawEvent.createMany({processed:false})` leaves orphans and creates no canonical Events; a scrape won't
   merge them. (Re-running the correct helper adopts a prior mis-run's orphans.)
3. **🔴 Schedule-rule (and any seed-data) CHANGES that must reach prod need a companion migration** — Vercel
   runs `migrate deploy`, never `db seed`. Mirror `kennels.ts`, make it idempotent, dry-run in a ROLLBACK txn.
4. **🔴 MEC platform (`?mec-events=`, Elementor):** static Cheerio; JSON-LD `startDate` **time is buggy**
   (parse displayed `.mec-event-time`/`.mec-start-time`); agenda year from the `mec-month-divider`; dedup
   duplicate skins by run number. (Platform note added to `source-platform-notes.md`.)
5. **🟢 Kennel `country` = full name** (`"United Arab Emirates"`), never the 2-letter code.
6. **Keep:** MEC title-filter, year-bearing dates, `upcomingOnly` + zero-row fail-loud, `title`-undefined
   synthesis, amber-palette grep-before-pick, bare-initialism alias discipline, self-host-logo (magic bytes),
   the `▶ FOR CLAUDE CODE` directive + ordered post-merge runbook, and triaging each review-bot finding on
   its merits (fix vs decline-with-reasoning + spawn follow-up) rather than auto-applying.
