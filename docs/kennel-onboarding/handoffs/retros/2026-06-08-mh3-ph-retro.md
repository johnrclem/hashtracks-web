## Handoff GAPS → research-prompt / platform-note improvements (the actionable part)

### A. 🔴 Google Sites can split words *inside* a label across spans — prefix-regex label matching is fragile

The first implementation matched labels with prefix regexes (`/sino\s*\(who\)/i`, `/saan\s*\(where\)/i`).
That works only because `stripHtmlTags` re-joins inline `<span>`s **without** inserting separators, so
`saan (whe`+`re)` → `saan (where)`. The **Codex adversarial review** flagged the latent failure: if
Google Sites ever emits a stray space *inside* a label word (`si no (who)`, `saan (whe re)`), those
regexes silently miss and **every scrape drops the hare + venue** while the date/run still parse — a
silent partial-data loss the fail-loud guard (which only checks the date) would never catch.

The live page was clean, so this never fired in production — but the robust shape is cheap and correct:

> **Platform change (done):** for Google Sites label blocks, **detect labels on a whitespace-collapsed,
> lowercased copy of the line** (`line.replace(/\s+/g,"").includes("sino(who)")`) and **read the value
> after the label's closing `)`** (`line.slice(line.indexOf(")")+1)`), rather than a label-prefix
> regex. This is immune to arbitrary intra-word span splits. Added a regression test feeding
> `si no (who)` / `saan (whe re)`.

### B. 🔴 Codacy runs `eslint-plugin-security` (local ESLint doesn't) — `Record[variable]` is an "object-injection sink"

The adapter's `ROMAN_VALUES[char]` and `MONTHS[key]` lookups (and string-index `s[i]`) tripped Codacy's
`detect-object-injection` ("Generic / Function Call Object Injection Sink") — **4 high-severity findings**
that local `npm run lint` showed **zero** of, because the repo's local ESLint config doesn't load the
security plugin. The clean fix isn't a suppression (the repo deliberately avoids per-line suppressions
on this rule — see `.codacy.yml`): it's to **use `Map.get()` instead of `Record`/array bracket access**
in new code. Converting `ROMAN_VALUES`/`MONTHS` to `Map`s and rewriting `romanToInt` as an index-free
right-to-left pass cleared all the object-injection sinks **and** fixed a Gemini-flagged out-of-bounds
`s[i+1]` access (safe under `noUncheckedIndexedAccess`) in one move.

> **Prompt/platform change:** when an adapter does keyed lookups into a constant map (month names, roman
> values, etc.) or indexed string/array access in a hot loop, **prefer `Map.get()` and `for…of` over
> `obj[var]`** — it's the difference between a clean Codacy run and 4 "high security" findings that
> local lint won't warn about. The two remaining Codacy hits (`detect-non-literal-html-method`:
> `stripHtmlTags(html)` / passing `html` to the parser) are the documented-accepted false-positive
> class every HTML adapter carries — left unsuppressed per `.codacy.yml`.

### C. 🟡 Codacy stale-PR cache froze on the pre-fix line numbers

After pushing the fix commit, the Codacy check stayed red with a **0-second run** and annotations
**pinned to the OLD commit's line numbers** (62/110/123/167) that no longer matched the refactored file
— the known `reference_codacy_stale_pr_issue_cache` behavior. The object-injection findings were
genuinely fixed (verified by `grep` for `obj[var]` sinks → none). **Don't churn commits chasing it** —
verify the code is clean, document, and (if a green tick is wanted) a maintainer clears the frozen check
from the Codacy dashboard. `main` is unprotected, so Codacy is non-blocking anyway.

### D. 🟢 `sitesv` logo: a `Referer` header fetched it server-side (refines the NSWHHH "always 403s" note)

The NSWHHH retro said Google Sites `lh3.googleusercontent.com/sitesv/…` logos 403 server-side → grab via
the browser. For Manila, **`curl -e https://sites.google.com/ <ogimage>` returned HTTP 200** (a real
200×155 PNG, magic-byte confirmed) — a `Referer` header was enough, no browser needed. The token still
rotates per page load, so browser-grab remains the reliable fallback, but a Referer'd `curl` is worth
trying first.

> **Platform change (done):** Google Sites `sitesv` note now says *"try `curl -e https://sites.google.com/`
> first; fall back to a browser grab if it 403s."*

### E. 🟡 Defensive roman-numeral fallback: kept despite a `/simplify` flag

The handoff requested a roman-numeral fallback for the run number (when the `=` decimal is absent).
`/simplify` flagged it as effectively dead (the live page always publishes the decimal). **Kept it**:
it parses *real redundant on-page data* (the page literally shows both `mmdccxxviii` and `= 2728`),
`runNumber` is a non-critical optional field, and removing it would contradict the handoff's explicit
ask. A genuine judgment call, not an oversight — noted so future reviewers don't re-litigate it.

## TL;DR for the research prompt + platform notes

1. **Google Sites label blocks: match labels on a whitespace-collapsed copy + read the value after `)`**,
   not a label-prefix regex — immune to intra-word span splits that would silently drop fields.
2. **Use `Map.get()` / `for…of`, not `Record[var]` / `arr[i]` / `str[i]`, in new adapter code** — Codacy's
   `eslint-plugin-security` flags bracket access as object-injection (local lint won't); `Map` also fixes
   `noUncheckedIndexedAccess`.
3. **Codacy stale-PR cache (0s run, old line numbers) → verify clean + document, don't churn.** Non-blocking.
4. **`sitesv` logos: try `curl -e https://sites.google.com/` first**; browser-grab is the fallback.
5. **Keep**: the `▶ FOR CLAUDE CODE` directive, Google Sites SSR (no browser-render), the
   single-block fail-loud guard, `upcomingOnly` for next-run-only pages, the 5-edit new-country
   `region.ts` pattern (incl. the **venue city** in `COUNTRY_INFERENCE_RULES`), slug-pin + bare-code
   alias omission for collision-prone codes, and `title` undefined → merge synthesis.
