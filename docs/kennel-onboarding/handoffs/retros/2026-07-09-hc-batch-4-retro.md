## What was NEW / CORRECTED this run (the durable lessons)

### New-1 · The handoffs' `COUNTRY_GROUP_MAP` edit was a DEAD no-op — only `STATE_GROUP_MAP` needs the metro
Both France handoffs (Lyon, Heraultics) instructed "add `"Lyon":"France"` / `"Montpellier":"France"`
to `COUNTRY_GROUP_MAP`." Reading `region.ts` disproved that: grouping is two-level —
`STATE_GROUP_MAP[metro] → group`, then `COUNTRY_GROUP_MAP[group] → country`. `COUNTRY_GROUP_MAP` is
keyed by **group name** (`"France":"France"`, `"United Kingdom":"United Kingdom"` already present), NOT
by metro. A metro key there is unreachable dead weight (the file's own comments say so: "metro 'Paris'
resolves via STATE_GROUP_MAP first, so only the state-group key 'France' is reachable here"). Correct
edit: **only `STATE_GROUP_MAP`** gets the 3 new metros; zero `COUNTRY_GROUP_MAP` edits. Reinforces
[[feedback_new_country_region_map_gotchas]] — the "COUNTRY_GROUP_MAP metro key is dead" rule.

### New-2 · France inference — add `montpellier|herault`, but OMIT bare `lyon` (the handoff's own caveat won)
The Lyon handoff proposed adding bare `lyon` to the France inference rule "with a disambiguation test"
— but a test asserting `inferCountry("Lyon County, Kansas") → USA` **cannot pass** if bare `lyon` is in
the rule (it'd match `\blyon\b` → France). The file's established discipline resolves this: bare
ambiguous city tokens that double as US place names are deliberately EXCLUDED (Warsaw, Belgrade,
Himalayan, Bombay all omitted for exactly this reason; `france` still catches "Lyon, France"). So:
**omit bare `lyon`**, add `montpellier` (double-L, unambiguous vs US single-L "Montpelier") + `herault`
(département) + accented `hérault`. Disambiguation tests assert `Montpellier, France → France`,
`Montpelier, Vermont → USA`, `Lyon County, Kansas → USA`. [[feedback_state_abbrev_collisions]]

### New-3 · A stale count is a floor, not a contract — trust the live freeze
titsh3's handoff estimated "~115 counted runs"; the freeze netted **113** after the `IsCountedRun=0`
drop of #1565 (and one HASH-CANCELLED guard). krashh3's handoff said "28 counted + 8 `#0` socials"; the
live feed held **36 total with 6 zero-numbered socials** (not 8). Both totals are the ground truth —
recorded the real numbers, didn't force the estimate. [[feedback_verify_handoff_backfill_feasibility]]

### New-4 · "0 upcoming" ≠ dead when the kennel ran yesterday
krashh3 returned **0 future events** from the HC future-only `getEvents` at build (its last run #32 was
the day before, biweekly Wed, no #33 posted yet). This is the *recently-active* case, not a failure —
the 36-run backfill supplies the page and the daily scrape adds #33 when posted. Shipped exactly as the
handoff predicted; the `upcomingOnly:true` guard protects the backfilled rows meanwhile.
